My project is a Disneyland Companion app. It's a chat bot that will help you plan out your day in Disneyland. Disney has a similar app called the Disney Genie, but when I tried to use it a few years ago, I was really disappointed in its functionality. It wouldn't let me add more than just a few must-do items, and didn't really offer the advice a first timer would be looking for. I wanted my project to do better.

This app has three main functions. The first functionality is to help you brainstorm your bucket list. This is especially important for first timers who may not know which rides or attractions are best suited for them. It then has functionality to plan your day's ride order to help maximize your time. Lastly, it has functionality to help you decide where and what you want to eat. This can help make sure that you don't miss the snacks you're interested in, and can help you sort through all the different dining locations. 

The architecture of my project uses multiple agentic principles. The main archictecture is a multi agent system. It has a main agent that will delegate the specific functions to sub agents as the user discusses what they want help with. There are three sub agents, each specialized for one of the functions listed above. 

The bucket list and foodie agents use skills primarily. One thing I wanted to accomplish with the skills was to give the agent all the context it needed to not hallucinate or miss things, without having to pay for that context over and over again. Using skills worked well here because it let the agent go through the general questions before loading the specifics. To use skills with these types of agents, I created a tool that takes in a file name and returns the content of that file to the agent. I then told the agent where their skill was located inside their system prompt. 

The last of my sub agents didn't use a skill, but managed a complex tool. This tool is a greedy algorithm to optimize wait times using projected wait times throughout the day. I vibecoded this function with Codex. One of the best parts of this vibecoding experience was getting Codex to ask me questions so I could think through how the function should act. The basis of the algorithm is that it has access to a dictionary of projected wait times of different rides at each half hour chunk of the day. It can then compare the 'current' wait time the algorithm is at to the average wait time of that ride to find what will save the most time. It also includes rough walking time between rides, and will weight rides in the same area more heavily than rides in other areas when determining the score. It takes in your bucket list for the day when determining what needs to be done, and will keep track of any rides that can't be completed in the given time. Using this repsonse, the agent can then talk with the user about tweaking the bucket list for the day. The agent also gets to decide inputs such as expected business and start-end times of the day. The agent also loads and formats the bucket list for input into the tool.

In addition to this tool, I use a few smaller tools. Each agent has tools that lets them save the information they generate (bucket list, schedule, food recommendations) to files for later access. The subagents have tools to talk to user (so they can communicate without ending the agent call) and a conclude function for when they want to end the agent call. I also gave them a talk to developer tool. It is the exact same thing as the talk to user, but the agent believes it is the developer in charge of them. I implemented this early on so that I could interact with the agents to know what they needed to perform well. This helped me catch bugs like when I hadn't added the right functions to my tool box. 

One of the last pieces of the architecture is agent memory. I let my main agent update a yaml file that saves some general user preferences. This supports some manner of logging in, or saving your details to come back and finish later. Each agent has the option to query this yaml file to get the user information, which means that the agents don't have to repeat questions. 

All of my agents had to undergo a lot of prompt engineering to ensure they did the right things in the right order. I had to update the prompts a lot for instructions on which questions to ask and in what order. In some instances, I had to give them additional information on how to interpret certain responses in order to function better. Some prompting went into making sure that they called the right tools at the right times, especially in the dynamic between talk-to-user, conclude, and just responding. 

One thing I learned while building this was to make sure you always understand the vision, even when you have Codex building parts of it. I had Codex write the entire optimization function, as well as writing most of my two skills. However, I was still engaged with Codex while building it. I told it what to include, how things should work, and what I wanted it to look like. Because I understood the vision, I could have Codex just do it, and still retain an understanding of how my application was working. 

The most difficult part of this was figuring out how to give my agents just enough information, without giving them too much or too little. For my bucket list agent, especially, I needed to make sure it knew enough about Disneyland to ask intelligent questions, but not too much that it compounded the context window and cost a lot of money. This is something I wish I had more time to work on. While it internally knows a lot about Disneyland, there are some specific things it would do, such as suggest Marvel character meet-and-greets while in Disneyland park (the Marvel characters are in the other park). I think one way I could've fixed this is by nesting skills. I first give it the option to call a general overview skill of a certain park, which gives it a few more details, or targeted question information, and then link that skill to the one with all the information.

Another lesson I learned was that your model matters. For most of my project, I defaulted to 5-nano because it was cheap, and because I previously hadn't seen a lot of issues with intelligence of 5.4 vs 5-nano. This was mostly sufficient for each of my subagents, although I would get problems with the agent sticking too closely to directions. My agents would just blindly keep following the questions list I had given it, not ever being creative with thinking about what question it should be asking next. There are some other ways I could fix this, but I think that upgrading them to 5.4 or something similar would help a lot. This is what I ended up doing with my main agent. As I was running tests to bring the system together, my main agent just felt really dumb. It was having trouble understanding responses back from the subagents and interfacing intelligently with the user. I changed this to the 5.4 model and never had any issues after that. I haven't found a lot of differences between models through many of the homeworks, but this showed that model choice really matters. I think I could've prompt engineered my way around it, but it would have taken a lot more time.

One thing that worked really well was getting my agents to interact with their tools. Between the instructions in the system prompt and the function descriptions, I really never had any issues with this. They would call the right tools with the right inputs basically every time. Writing a lot of these functions became very easy with Codex helping.
