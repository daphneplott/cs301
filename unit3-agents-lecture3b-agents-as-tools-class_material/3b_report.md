The first idea I had for the gaurdrails was to design two agents. One of them would be a basic agent, and it would function as a tool for the main agent. The main agent would be in charge of screening the input and output, and communicating with the basic agent. 

However, as I was starting to implement it, I realized I was going to have problems keeping the chat bot's memory, so I decided to switch the roles. I gave the chatbot two functions, one to screen the input and offer emphasis about taboo topics, and one to screen the output and give feedback. As I was writing it, I had a hard time getting the chat bot to call the input screener. I think the problem was that the prompting for that was much less significant than the output screener. I wonder if the agent thought that it was just fine without an input screener. One thing that may have been happening was that it kept calling the output gaurdrails for the input because the function names were too similar. Changing this helped it call the input screener more often.

One thing I liked about this system was that I felt like the secret keeping was better when I could have one agent with each goal. I felt like it was the equivalent of talking something out with someone else because they usually are thinking about things differently than you, and they can catch what you miss. One thing that this really helped with was indirect references. My taboo topic was rubber ducks, and one thing I tried to get it to do was finish the line from a Harry Potter book about rubber ducks. The first model was very willing to do so, but the gaurdrails gots that it was talking about rubber ducks. However, it did cost a lot more money, even with the nano versions. Simple conversations were now taking almost a cent. I think part of the issue was how much the messages were repeated in function calls.

One interesting thing that happened was that sometimes the chat would call the guardrails when it asked its initial question, "What's on your mind today?", and sometimes it wouldn't. Sometimes the gaurdrails would just approve it, but once or twice it got really confused because it wasn't a 'real' output.

I was trying to implement the functionality of having the bot talk about certain topics if they were given the password. However, I forgot to let my gaurdrails know that was a possibility, so they bullied my chat into not talking about it. As I was thinking how to implement this, I wanted to think about how to do so safely. I didn't want to just tell the guardrails about the password clause in case the chat model thought it could fake the user giving it the password. What I did was give the guardrails a separate function that let it talk to the user to get the password.

One issue I started to run into was that my chat bot didn't always output the exact thing it gave to my gaurdrails. To fix this, I changed the talk to user function so that my chat read incoming messages, but my guardrails was the only one that could output messages. It mostly worked out this way, but I ran into a couple issues. My first issue was that my gaurdrails didn't realize that "How can I help you?" was the response my bot wanted to send, and would get confused, not talk to the user, and try to respond on its own. My next problem was that my chat tried to respond on its own, without using the given pathways, and ended up closing the chat thread. I ran out of time after that, but I wonder if it would be better to rename the guardrails function to be something like "output to user" to help my chat understand better the flow.
